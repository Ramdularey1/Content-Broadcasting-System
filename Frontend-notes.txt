Project structure
- src/components: reusable layout, UI, table, form, modal, toast, and state components.
- src/context: auth and content providers for cross-page state without prop drilling.
- src/services: auth, content, and approval service modules. Components do not call APIs directly.
- src/pages: route-level screens for auth, teacher, principal, and public live flows.
- src/utils: formatting, schedule status, and seeded mock data helpers.

Authentication flow
- Login calls auth.service.js and stores a mock token plus user profile in localStorage.
- AuthProvider hydrates the session on page load.
- ProtectedRoute redirects unauthenticated users to /login.

Role-based routing
- Teacher routes only allow the teacher role: dashboard, upload, and my content.
- Principal routes only allow the principal role: dashboard, approvals, and all content.
- Public live route is intentionally open and requires no session.

API integration approach
- All async operations go through services/auth.service.js, services/content.service.js, or services/approval.service.js.
- The current services simulate latency, failure, broken responses, filtering, and list data.
- Replacing the API means swapping service internals for fetch or Axios while keeping provider/page contracts stable.

State management approach
- AuthProvider manages session state.
- ContentProvider manages content list, loading, error, upload, approve, reject, and refresh actions.
- Derived dashboard counts and filtered rows are memoized where useful.

Assumptions
- File upload is frontend-only and previews object/data URLs.
- Scheduling is UI-based and calculates scheduled, active, or expired status from content start/end times.
- Public live page shows approved active content for the provided teacher id and polls periodically.
- Mock data includes a larger generated list to demonstrate scalable rendering, filtering, and pagination.
